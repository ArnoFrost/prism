# Review-Merge 规范

> 从 workflow-review Merge 阶段提取的独立规格。定义多角色评审结果的合并逻辑。

## 适用条件

- ReviewSpec.topology = `parallel`（多角色独立评审）
- 角色数 >= 2
- topology = `serial` 时本规范不适用（无需合并）

## 合并流程

```
角色报告 ×N
    │
    ▼
① 发现去重
    │
    ▼
② 冲突仲裁
    │
    ▼
③ 独立发现率计算
    │
    ▼
④ 统一行动计划
    │
    ▼
综合报告
```

## ① 发现去重

| 场景 | 处理方式 |
|------|---------|
| 多角色发现内容实质重叠 | 保留证据最充分的版本，标注发现来源角色 |
| 描述不同但指向同一根因 | 合并为单条，注明多角色均发现 |
| 完全独立的发现 | 全部保留 |

判定标准：两个发现指向同一代码位置/文档段 **且** 建议的行动方向相同 → 视为重叠。

## ② 冲突仲裁

| 冲突类型 | 仲裁规则 |
|---------|---------|
| 优先级分歧（如 A 判 P1，B 判 P0） | 按影响范围裁决：安全/阻塞 → P0 > 功能缺陷 → P1 > 规范 → P2 |
| 行动方向冲突（如 A 建议删除，B 建议重构） | 综合报告中列出双方立场 + 仲裁结论 + 理由 |
| 范围分歧 | 以 scope.md 为准，超出 scope 的发现降级为 P2 或 Open Question |

## ③ 独立发现率

```
独立发现率 = |非重叠发现| / |总去重发现| × 100%

非重叠发现 = 仅由单个角色提出的发现
总去重发现 = 合并后的唯一发现总数
目标 >= 50%
```

综合报告中**必须**包含计算表格：

| 角色 | 总发现 | 仅由该角色发现 | 独立项数 |
|------|--------|--------------|---------|
| A | — | — | — |
| B | — | — | — |
| C | — | — | — |
| **合计** | — | — | 独立发现率: —% |

独立发现率 < 50% 时应附注："角色分工可能重叠过多，建议下次调整角色 Scope"。

## ④ 统一行动计划

合并后输出统一行动计划表：

| # | 行动 | 来源角色 | 优先级 | Owner | 验收标准 |
|---|------|---------|--------|-------|---------|
| 1 | ... | A, B | P0 | ... | ... |

- 去重后按 Owner 分组
- 优先级以仲裁后结果为准
- 来源角色标注用于追溯

## 与 workflow-review 的关系

本规范对应 workflow-review 骨架中的 **Merge 阶段**（步骤 1-2）。

- review SKILL.md 的"合并规则"段引用本规范
- ReviewSpec.reduction_rule 字段的取值由本规范约束
- workflow-review-lite 不适用本规范（topology=serial，无 merge）
