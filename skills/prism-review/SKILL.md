# 多角色协作评审 (Prism Review)

> 以"总分总"结构组织多角色独立评审，综合仲裁后输出统一行动计划

## 触发条件

| 条件 | 示例 |
|------|------|
| 评审方案/规范/代码 | "评审这个技术方案"、"Review 这份规范" |
| 多角色/多视角评审 | "从不同角度评审"、"双视角评审" |
| 用户明确要求 | "启动评审"、"/prism-review" |

## 参数

| 参数 | 可选值 | 默认 | 说明 |
|------|--------|------|------|
| `--mode` | `full` / `quick` | `full` | full = 多文件 + 并行 subagent；quick = 单文件串行 |
| `--depth` | `surface` / `deep` | `deep` | surface = 快速扫描；deep = 深度审查含代码级引用 |

用户未指定时根据评审对象规模自动判断。

## 协作骨架：总分总 (Align → Explore → Merge)

```
┌─────────────────────────────────┐
│  1. Align（对齐）                │
│  - 确认评审对象、范围、成功标准  │
│  - 选定角色组合与关注优先级      │
│  - 选择执行策略（并行/串行）     │
├─────────────────────────────────┤
│  2. Explore（独立评审）          │
│  各角色独立输出评审章节          │
│  角色之间禁止互相引用            │
├─────────────────────────────────┤
│  3. Merge（综合仲裁）            │
│  - 合并去重、冲突仲裁            │
│  - 独立发现率自检                │
│  - 优先级排序、统一行动计划      │
└─────────────────────────────────┘
```

## 默认角色（3 角色）

用户可根据场景增减（上限 5 个）。

### 角色 A：结构与一致性评审员

- **Identity**: 结构与一致性评审员
- **Scope**: 目录结构、命名规范、入口文件完整性、引用一致性、Single Source of Truth
- **Anti-patterns**: 不评价业务逻辑正确性，不讨论实现细节

### 角色 B：可执行性评审员

- **Identity**: 可执行性评审员
- **Scope**: 行动项可落地性、验收标准明确性、依赖识别、优先级合理性、MVP 边界
- **Anti-patterns**: 不纠结格式细节，不做架构重设计

### 角色 C：风险与边界分析师

- **Identity**: 风险与边界分析师
- **Scope**: 安全风险、范围漂移、过度工程化、向后兼容、依赖风险、滥用可能
- **Anti-patterns**: 不提改进方案（那是行动项的事），只识别风险和边界问题

### 自定义角色

每个角色定义须包含 Identity / Scope / Anti-patterns 三字段。

## 输出契约

| 字段 | 说明 | 必需 |
|------|------|------|
| **TL;DR** | 结论摘要，<= 3 句 | 是 |
| **Findings** | 客观发现，按 P0/P1/P2 分级 | 是 |
| **Risks** | 未来影响预判（概率/严重度/缓解建议） | 是 |
| **Actions** | 行动项（Owner/优先级/验收标准） | 是 |
| **Prior Unclosed Items** | 上次评审未关闭的行动项复检 | 是（re-review） |
| **Open Questions** | 未决问题 | 按需 |

### Findings 分级标准

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| **P0** | 阻塞级 | 安全问题、核心逻辑错误、阻塞后续工作 |
| **P1** | 重要 | 功能缺陷、设计不一致、用户体验显著影响 |
| **P2** | 改善 | 规范对齐、体验优化、可延后处理 |

## 执行策略

### 策略一：并行子任务（推荐，mode=full）

为每个角色启动独立子任务，实现真正的上下文隔离。

**Align 阶段（主 Agent）：**
1. 执行项目探测（检查 `workspace.*.local` 或 `ai-task.local`）
2. 确认评审对象、范围、角色选择
3. 打包 `context_payload` 给子任务

**Explore 阶段（并行子任务）：**
在同一轮响应中为每个角色发起独立子任务。子任务 prompt 包含角色定义、评审对象、输出要求。

**Merge 阶段（主 Agent）：**
1. 按合并规则去重仲裁
2. 计算独立发现率（目标 >= 50%）
3. 输出统一行动计划
4. 按产物规则落盘

### 策略二：串行角色切换（mode=quick）

在单次会话中依次以不同角色输出，完成后执行 Merge。

### 项目探测（Align 阶段）

检查工作仓库是否存在 Prism 工作区：

| 检测目标 | 行为 |
|---------|------|
| `workspace.*.local/` | 读取 project.yaml、index.md、README.md 作为背景上下文 |
| `ai-task.local/`（兼容） | 同上 |
| 均不存在 | 按通用模式执行 |

## 产物设计

### mode=full：多文件产物

```
{output_dir}/
├── task_plan.md              # Merge 综合报告
├── reviewer_A_{角色名}.md     # Agent A 独立报告
├── reviewer_B_{角色名}.md     # Agent B 独立报告
├── reviewer_C_{角色名}.md     # Agent C 独立报告
└── (可选) mermaid/
```

### mode=quick：单文件产物

所有角色评审 + Merge 输出到一个 markdown 文件中。

## 合并规则

| 场景 | 处理方式 |
|------|---------|
| 多角色发现重叠 | 保留证据最充分的版本 |
| 优先级冲突 | 按影响范围裁决：P0 > P1 > P2 |
| 行动项合并 | 去重后按 Owner 分组 |

## 反馈闭环

1. **质量自检**：输出独立发现率指标
2. **Calibration 询问**：向用户确认发现价值
3. **增量 Re-review**：支持 `--incremental` 只审查 diff 部分
